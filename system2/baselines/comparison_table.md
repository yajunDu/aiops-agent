# 论文 4.2 节 - 方法对比表（真实数据）

## 表 4.X 根因诊断准确率（Acc@1）对比

| 方法 | 整体 Acc@1 | CPU 故障 | NETWORK 故障 | POD_KILL 故障 | 平均耗时 |
|---|---|---|---|---|---|
| Rule-Based | 64.9% | 96.8% | 10.0% | 83.3% | 0.1s |
| MicroRCA | 43.9% | 48.4% | 30.0% | 66.7% | 0.5s |
| Naive LLM+RAG | 59.6% | 100.0% | 10.0% | 16.7% | 0.8s |
| **Ours (Dual-Process)** | **82.5%** | 100.0% | 55.0% | 83.3% | 4.7s |


## 论文 4.2 节定稿话术

> 本研究将所提双过程认知架构与三类经典方法对比：
> 
> 1. **Rule-Based**: 基于阈值规则的传统方法，整体 Acc@1 仅 64.9%，在 NETWORK 故障上仅 10%——规则方法无法编码 CNI 层"流量正常但延迟异常"的灰色信号。
> 
> 2. **MicroRCA**: 基于 PageRank 因果图传播的方法 [Wu et al. 2020 ICSOC]，整体 Acc@1 43.9%。在包含 24 个边缘服务持续 CrashLoop 的嘈杂集群环境中，PageRank 被环境噪声扰动，反而比纯规则更差——印证了**因果图传播需要语义层面的"信号 vs 噪声"区分能力**，这正是 LLM 的核心优势。
> 
> 3. **Naive LLM + RAG**: 用同样的 Qwen2.5-7B-AWQ 模型，把所有指标摘要一次性塞进 prompt 但不调用工具。整体 Acc@1 59.6%，POD_KILL 仅 16.7%——说明**LLM 单纯阅读文本摘要时关键信号（如 restart_jump）容易被淹没**，需要主动通过 tool calling 聚焦关键指标。
> 
> **本文方法**（系统1 孤立森林 + 系统2 LLM Agent + GraphRAG）：整体 Acc@1 **82.5%**，相比三基线分别提升 +17.6pp / +38.6pp / +22.9pp。三类故障中 CPU 100%、POD_KILL 83.3% 均达到基线最优水平，NETWORK 55% 显著超越所有基线——证明**双过程架构 + 工具调用 + 图谱推理的协同效应**远胜于任一单一方法。

## 关键洞察

1. **同等 LLM 模型下**，本文方法 vs Naive LLM = 82.5% vs 59.6%（+22.9pp），证明改进**不来自模型规模**，而**来自方法论本身**（双过程 + 工具调用 + GraphRAG）。

2. **复杂环境韧性**：在 24 个边缘服务持续 CrashLoop 的告警风暴背景下，本文方法仍能准确诊断目标故障；而 MicroRCA 几乎瘫痪（43.9%）——这是工业级部署的关键能力。

3. **NETWORK 故障是分水岭**：三基线 10-30%，本文 55%，差距集中在网络场景。这与论文 4.3.1 节"CNI 层观测盲区"的论述呼应——LLM 通过工具调用可以**反复尝试不同的指标视角**（net_rx → net_tx → throttle → restart）才能拼凑出网络故障的全貌。
